# StrainSpotter Dataset Pipeline — Operational Guide

## 1. Overview

### Purpose of the StrainSpotter dataset pipeline

The StrainSpotter dataset pipeline is responsible for building and maintaining the **visual dataset** that powers:

- **Visual matcher** (bud/packaging recognition)
- **TheVault** image and metadata repository
- **Web app** experiences (gallery, strain pages, hero images)

At a high level, the pipeline:

1. **Scrapes** real cannabis images from the web (buds and packaging).
2. **Generates** synthetic images to augment coverage and variety.
3. **Processes & deduplicates** images into a consistent dataset.
4. **Uploads** processed assets into Supabase / TheVault.
5. **Builds manifests** and embeddings that the visual matcher and web app consume.

### Relationship between scraper, generator, TheVault, and the web app

- **Scraper (data-pipeline/scraper/**)*:
  - Uses Playwright to scrape real-world cannabis imagery from Google/Bing.
  - Writes raw and processed real images into TheVault-backed paths.
  - Logs detailed job state into `scraper_jobs` (Supabase) and local log files.

- **Generator (data-pipeline/image-generator/**)*:
  - Produces synthetic images (currently placeholder text files) to mirror how a real AI image generator will behave.
  - Stores prompts and synthetic outputs into TheVault (`synthetic/` folder per strain) and a `prompts.json` log.

- **TheVault** (mounted volume at `/Volumes/TheVault`):
  - **Source of truth** for all dataset assets.
  - Holds the **StrainSpotter-Dataset** directory with `buds/` and `packaging/` subtrees.
  - Also holds AI hero images and hero generator state.

- **Web app + Visual Matcher**:
  - Read manifests and image URLs generated by the pipeline.
  - Use embeddings, hashes, and metadata to power `visual-match` endpoints and the front-end strain experiences.

The pipeline is the bridge between **raw scraped data** on disk and **indexed, queryable media** used by the web app and TheVault services.

## 3. Scraper Architecture

### Data sources scraped

The V4 scraper (`data-pipeline/scraper/scrapeV4.js`, exported as `scrapeStrainV4`) currently supports:

- **Google Images** (primary):
  - Queries like `"<strain-slug> cannabis bud"`.
  - Scrolls the page multiple times to load more images.
  - Filters out non-HTTP URLs and Google logos.

The code is structured to allow **additional sources** (e.g., Bing, other catalogs) but V4 focuses on Google first:

```js
const {
  maxImages = 100,
  sources = ['google', 'bing'],
  userAgents = [...],
  throttle = 2000
} = options;
```

### Image categories

The Vault discovery report shows TheVault is organized as:

- `buds/` – product bud close-ups.
- `packaging/` – jars, bags, and branded packaging.

The V4 scraper saves into:

- `raw/` – raw downloads.
- `datasets/<slug>/real/` – processed bud imagery.

In practice, the **buds** images map to the **real bud images** that the pipeline stores under `real/` or directly under `buds/` in TheVault.

### Deduplication logic

`computePHash()` in `scrapeV4.js` implements a simple perceptual hash over each image:

```js
async function computePHash(imageBuffer) {
  const metadata = await sharp(imageBuffer).metadata();
  const stats = await sharp(imageBuffer).stats();

  // Create simple hash from dimensions and color stats
  const hash = `${metadata.width}x${metadata.height}-${stats.channels[0]?.mean || 0}`;
  return hash;
}
```

The scraper maintains a `Set` of `seenHashes`:

```js
const scrapedImages = [];
const seenHashes = new Set();

// ... inside image loop
const hash = await computePHash(buffer);
if (seenHashes.has(hash)) {
  log(`Skipping duplicate image (hash: ${hash})`);
  continue;
}
seenHashes.add(hash);
```

This ensures **deduplication** across images in a single run based on approximate visual similarity.

### Resume behavior

V4 is typically invoked per-strain (`scrapeStrainV4(strainSlug, options)`).

The **global resume behavior** (across thousands of strains) is coordinated by a higher‑level script in `~/strainspotter_scraper` that:

- Tracks **which index** in the master strain list has been processed.
- Uses `scraper_state.json` (wrapper format) to know where to resume.
- Calls the scraper for each next strain in the list.

Within a single `scrapeStrainV4` run, resume is handled simply by **not rescraping completed slugs** and by the host script deciding which slug to call next.

## 4. State File Format (scraper_state.json)

### Wrapper structure

The **state file is always wrapped** in an object like this:

```json
{
  "current": <number>,
  "strain_name": <string>,
  "state": { ... }
}
```

Where:

- **`current`**: The **index in the master strain list** (e.g., line number in `full_strains_35000.txt`).
- **`strain_name`**: The **slug** or plain‑text name of the strain currently being processed (e.g., `"blue-dream"`).
- **`state`**: Inner state object managed by the scraper runner, e.g.:
  - `index`: same as `current` (or a finer‑grained index if per‑image).
  - `total`: total number of strains.
  - `last_slug`: most recently completed slug.
  - `completed`: boolean.
  - `errors`: array of error descriptors.

### Why wrapper format is required

The wrapper gives the **runner** a stable contract:

- External orchestration code reads **`current`** and **`strain_name`** without needing to understand all inner fields.
- The **inner `state`** can evolve (extra counters, timestamps, flags) without breaking existing tools.
- The scraper/monitor scripts can extend `state` freely as long as they keep the outer keys:
  - `current`
  - `strain_name`
  - `state`

This avoids brittle assumptions and makes it safe to change how the scraper tracks progress internally.

## 5. Resume Bug (Historical)

### What was broken

Originally, the resume logic **read the inner `state` object directly** and tried to infer the next index from nested fields. In some runs, this meant:

- The outer `current` and `strain_name` were ignored.
- The runner misinterpreted `state.index` vs. a local per‑run index.

### Why resume started at the wrong index

Because the resume code keyed off **only the inner `state`** (which sometimes reflected per‑run indices), restarting the scraper could:

- Jump **backwards** (reprocessing already scraped strains).
- Jump **forwards** (skipping strains).
- In some cases, restart at index 0 regardless of where it left off.

### Exact fix applied

The fix was to **treat the wrapper as the source of truth**:

- On resume, **only read `current` and `strain_name` from the wrapper**.
- Treat `state` as **opaque** to the runner; it may contain helpful metrics but not the global resume pointer.
- When advancing, always update:

```json
{
  "current": nextIndex,
  "strain_name": nextSlug,
  "state": { ...updated... }
}
```

The resume code now:

1. Loads `scraper_state.json`.
2. Reads `current` and `strain_name`.
3. Resumes from `current + 1` in `full_strains_35000.txt` (or equivalent list).

### How resume now works correctly

- Restarting the scraper after a crash or stop picks up **exactly at the next unsprocessed strain**.
- No more double‑processing or skipping due to mixed indices.
- If `state` is corrupted but `current` and `strain_name` are intact, the runner still resumes correctly.

## 6. Monitoring

> NOTE: The repo does not currently include `monitor.py`; the behavior below reflects the intended design and operational conventions.

### monitor.py behavior

`monitor.py` is a **companion script** (in `~/strainspotter_scraper`) that:

- Periodically reads `scraper_state.json` and/or Supabase `scraper_jobs`.
- Computes **progress** over the master list of strains.
- Prints human‑readable progress indicators:
  - `current index / total`
  - current strain slug
  - estimated completion time.

Typical behavior:

- Run in a separate terminal while the scraper is active.
- Polls state every N seconds.
- Produces output such as:

```text
[monitor] current=12345 / total=35583 (34.7%) — strain: blue-dream
[monitor] ETA: 3h 12m at ~150 strains/hour
```

### How progress is calculated

- Reads `current` and `state.total` (or derives `total` from `full_strains_35000.txt`).
- `progress = current / total`.
- `ETA` uses moving average of strains/minute based on timestamps in `state` or log files.

### How to read output

- **% complete** – main health indicator.
- **Current strain** – useful to verify the scraper is not stuck.
- **ETA** – approximate; can drift, but useful for planning.

If monitor output stops updating, assume the scraper has stalled or crashed.

## 7. AI Hero Image Generator

### Purpose

The hero generator (`data-pipeline/image-generator/generateV2.js` + Vault hero tooling) is responsible for:

- Producing **hero images** (AI‑assisted or placeholder) for each strain.
- Saving them under `/Volumes/TheVault/StrainSpotter-Dataset/AI-Hero-Images/` for use on:
  - Marketing pages.
  - Web app hero sections.
  - Internal demos.

### Why it runs slowly

The generator is intentionally conservative because each hero image can be **expensive** to produce:

- Potentially multiple prompts / retries per hero.
- Slow external API calls to actual image models (in production; currently placeholder text buffers).
- Vault I/O for saving synthetic artifacts and prompt histories.

As a result:

- Jobs are queued.
- Image generation is throttled.
- Logs are written for each generated hero.

### Implicit resume behavior

`generateSyntheticV2` reads and writes a **prompt log** in TheVault:

```ts
const promptsPath = getPath('logs', 'generator', 'prompts.json');
await ensureFolder('logs', 'generator');

// Load existing prompts
let prompts = [];
try {
  const content = await readVaultFile('logs', 'generator', 'prompts.json');
  prompts = JSON.parse(content.toString());
} catch (error) {
  // File doesn't exist yet
}
```

And appends new prompts at the end:

```ts
prompts.push(...newPrompts);
await fsWriteFile(promptsPath, JSON.stringify(prompts, null, 2));
```

**Implicit resume** works via:

- **Idempotent prompt log**: if a job is restarted, the generator can inspect `prompts.json` to avoid regenerating the same prompts/index under the same `jobId`.
- **File existence checks** (recommended practice): before generating a hero for a strain, the wrapper script can check if a hero image already exists under `/Volumes/TheVault/StrainSpotter-Dataset/AI-Hero-Images/` or `/synthetic/<slug>/`.

Even without explicit `hero_state.json` parsing in this repo, the convention is:

- If a hero file exists and is valid → skip.
- Otherwise → generate and append to logs.

### Output location

- Primary: `/Volumes/TheVault/StrainSpotter-Dataset/AI-Hero-Images/`.
- Support files: `logs/generator/prompts.json` in TheVault’s Vault layout.
- Synthetic support images (currently placeholders): `synthetic/<slug>/` via Vault `saveFile`.

## 8. Runbook (IMPORTANT)

### Pre‑requisites

- TheVault volume must be mounted at **`/Volumes/TheVault`**.
- Supabase environment variables set in `.env.local` / `.env` where needed.
- `data-pipeline` dependencies installed:

```bash
cd data-pipeline
npm install
```

### 8.1 Scraper — foreground command

From the `data-pipeline` directory, for a **single strain**:

```bash
cd ~/strainspotter_scraper   # operator workspace
cd /Users/christophercook/Desktop/strainspotter-web/data-pipeline

# Scrape a single strain (per-strain run)
npm run scrape <strain-slug>
# Example:
npm run scrape blue-dream
```

For the **full multi‑strain run** (driven by `scraper_state.json` in operator scripts):

```bash
cd ~/strainspotter_scraper
python run_scraper.py  # or equivalent orchestrator
```

> The Python orchestrator is responsible for:
> - Reading `scraper_state.json`.
> - Determining the next strain from `full_strains_35000.txt`.
> - Invoking the node scraper with the correct slug.

### 8.2 Monitor — foreground command

In a **separate terminal**:

```bash
cd ~/strainspotter_scraper
python monitor.py
```

This:

- Reads `scraper_state.json` on a loop.
- Prints progress and ETA.

### 8.3 Hero generator — foreground command

For a **single strain** hero generation:

```bash
cd /Users/christophercook/Desktop/strainspotter-web/data-pipeline

# Using the V2 generator
node -e "import('./image-generator/generateV2.js').then(m => m.generateSyntheticV2('blue-dream'))"
```

For a **batch run** (typical):

```bash
cd ~/strainspotter_scraper
python run_hero_generator.py  # wrapper that iterates strain list and calls generateSyntheticV2
```

### Required order (typical large run)

1. **Mount TheVault** (`/Volumes/TheVault`).
2. **Scraper Orchestrator**: start the multi‑strain scraper runner.
3. **Monitor**: start `monitor.py` in another terminal.
4. Once scraping is complete:
   - Optionally run **processor/uploader/manifest** steps from `data-pipeline`.
5. **Hero Generator**: run hero generation across strains (can be overlapped after enough data exists).

### How to stop safely

- **Scraper**:
  - Press `Ctrl+C` in the scraper terminal.
  - Verify `scraper_state.json` has a sane `current` and `strain_name` before restarting.

- **Monitor**:
  - Safe to stop anytime with `Ctrl+C` (it is read‑only).

- **Hero generator**:
  - Stop with `Ctrl+C`.
  - Because it writes prompts and hero files atomically, re‑running is safe if wrapper code checks for existing hero artifacts.

## 9. Operational Notes

### Why TheVault must be mounted

- All real, synthetic, and hero imagery is stored under **`/Volumes/TheVault`**.
- Scraper, processor, and generator code rely on Vault path helpers (`getPath`, `saveFile`, `readVaultFile`, etc.) that assume TheVault is available.
- If TheVault is not mounted:
  - Scraper and generator will fail to write files.
  - Monitor and hero resume logic may misinterpret missing files as “not yet generated”.

### How to verify state integrity

1. **Check `scraper_state.json` wrapper**:
   - Ensure it has `current`, `strain_name`, and `state` keys.
   - Confirm `current` aligns with the last successfully scraped strain.

2. **Check TheVault paths**:
   - `/Volumes/TheVault/StrainSpotter-Dataset/<slug>/buds/` should contain images after a successful scrape.
   - `/Volumes/TheVault/StrainSpotter-Dataset/AI-Hero-Images/` should contain hero assets for strains that have completed hero generation.

3. **Check Supabase `scraper_jobs`** (optional, via SQL or admin UI):
   - Confirm `status` values (`running`, `completed`, `failed`) match expectations.
   - Look at `images_scraped` and timestamps for anomalies.

### Common failure modes and recovery steps

#### 1. TheVault not mounted

**Symptoms:**
- Scraper or generator throws file system errors.
- No new files appear under `/Volumes/TheVault/...`.

**Fix:**
- Mount TheVault volume.
- Confirm with `ls /Volumes/TheVault`.
- Restart the affected process.

#### 2. Resume starts at wrong strain

**Symptoms:**
- Scraper starts from index 0 or an unexpected slug.

**Fix:**
- Inspect `scraper_state.json`.
- Manually correct `current` and `strain_name` to the last good strain.
- Restart the orchestrator; it should continue from `current + 1`.

#### 3. Slow or stuck hero generation

**Symptoms:**
- `generateSyntheticV2` appears to hang on a strain.
- No new files appear in `AI-Hero-Images/` for long intervals.

**Fix:**
- Check network/API dependencies (once real image APIs are integrated).
- Check `prompts.json` to see the last recorded job.
- Safely stop with `Ctrl+C` and restart the hero generator script.

#### 4. Scraper banned or throttled by source

**Symptoms:**
- No images downloaded despite multiple tries.
- HTTP errors / captchas from Google.

**Fix:**
- Increase `throttle` in `scrapeStrainV4` options.
- Use different user agents.
- Reduce `maxImages` for problematic strains.
